# Phase 6 — UI Design Contract: Auth & Shell

> Visual and interaction contract for Phase 6 (Auth & Shell). Phase 5 produced the design-token spine; this document **adds screen-level contracts** for login, invitation/reset flows, the authenticated app shell, the admin URL display modal, the user menu, the error boundary, and the 404 page — all on top of the unchanged Phase 5 token layer.

## Scope Clarification

Phase 6 introduces these new visible surfaces:

1. `app/(public)/login/page.tsx` — minimal-layout login form
2. `app/(public)/invite/[token]/page.tsx` — set initial password from invitation URL
3. `app/(public)/reset/[token]/page.tsx` — set new password from reset URL
4. `app/(public)/login/page.tsx` (success/expired token landing flows)
5. `app/(authed)/layout.tsx` — full app shell wrapping every authenticated route
6. `app/(admin)/[adminSegment]/layout.tsx` — admin shell (same chrome as authed, plus role gate)
7. `<Topbar>` component — logo + page-title slot + spacer + locale toggle + theme toggle + user menu
8. `<UserMenu>` component — display name + caret → dropdown with Logout
9. `<InviteUrlModal>` / `<ResetUrlModal>` component — admin-side "URL displayed once" modal with copy button (UI primitive used by Phase 9 admin pages; ships in Phase 6 because it is an auth flow primitive)
10. `app/error.tsx` — generic error boundary (FR + EN)
11. `app/not-found.tsx` — 404 page (FR + EN)
12. Sonner toast wiring for: logout success, invitation/reset success, network/server errors

Phase 6 does NOT redefine any token from Phase 5. It only declares:
- New components and their composition rules
- Screen-level layout patterns (centered card, dropdown menu)
- The full v10 i18n key port + Phase 6-specific new keys
- Phase 6 copywriting (FR + EN) for every new user-facing string

## Design System

| Property | Value | Source |
|----------|-------|--------|
| Tool | none (no shadcn) | inherited from Phase 5 — Tailwind v4 CSS variables only |
| Preset | not applicable | — |
| Component library | none in Phase 6 | dropdown / modal use vanilla React patterns; Radix primitives may be considered but are NOT introduced in Phase 6 (avoid scope creep — see Component Strategy below) |
| Icon library | lucide-react (Phase 5) | new icons added in Phase 6: `LogOut`, `User`, `Copy`, `Check`, `AlertTriangle`, `Eye`, `EyeOff` |
| Font | Plus Jakarta Sans (Phase 5) | unchanged |

**Component strategy for Phase 6:**

Two new interactive primitives are required: a **user-menu dropdown** and a **modal dialog**. Both are implemented as plain React + Tailwind (no Radix, no shadcn) for these reasons:

- The user menu has only 1 trigger and 1 logout item — Radix `<DropdownMenu>` is overkill; a simple click-outside hook + Escape-key listener suffices.
- The invite/reset URL modal is shown once per admin action and contains a single copy button + dismiss — Radix `<Dialog>` adds 30+ kB for behaviour we can implement in ~40 lines.
- Phase 5's "no shadcn, no Radix" rationale (small surface, 2-3 core screens, custom v10 visual language) still holds.
- If Phase 9 (admin partner-listing) needs richer modal behaviour (form modals, confirmation dialogs), Radix Dialog can be introduced then — Phase 6's modal stays minimal.

**Accessibility floor (NON-NEGOTIABLE for both primitives):**
- Trap focus inside the open menu/modal
- Restore focus to the trigger on close
- Close on Escape
- Close on outside click (menu) / explicit close button (modal)
- `role="menu"` / `role="dialog" aria-modal="true"` + `aria-labelledby`
- Keyboard navigation: arrow keys for menu items; Tab inside dialog cycles
- All triggers reachable via Tab; visible focus ring (Phase 5 teal box-shadow)

**Registry safety gate:** not applicable — no shadcn, no third-party blocks. This is recorded in §Registry Safety below.

### Invite / Reset Page Layout (`app/(public)/invite/[token]/page.tsx`, `app/(public)/reset/[token]/page.tsx`)

Same skeleton as login (centered card, top-right toggles, bottom footer) with these differences:

- **Card title:** "Définir votre mot de passe" / "Set your password" (invite) or "Réinitialiser le mot de passe" / "Reset your password" (reset)
- **Card max-width:** 480px (slightly wider than login)
- **Form fields:**
  1. New password — type=password, with eye-toggle button on right (Lucide `Eye` / `EyeOff`, 17px, color `--muted`, click toggles type=text/password)
  2. Confirm password — type=password, same eye-toggle
- **Inline strength feedback:** below the new-password field, a row of 4 segments (8px tall, 4px gap, 6px border-radius). Filled segments use `--gd`; empty use `--border`. Labels: "Faible/Moyen/Fort/Très fort" / "Weak/Medium/Strong/Very strong". Source rule: zxcvbn-style scoring is NOT required — a simple length-and-class check is sufficient (≥8 chars + at least one number + one upper-case = "Strong"). The planner picks the exact rule; this UI-SPEC requires the segments + label only.
- **Mismatch error:** if confirm ≠ new, the confirm input gets `.invalid` class (`--danger` ring) on blur, and an inline message under the field reads "Les mots de passe ne correspondent pas." / "Passwords do not match."
- **Submit button:** "Définir le mot de passe" / "Set password" — `.btn-green`, full-width
- **Expired/used token state:** if the page loads with an invalid/expired/used token, replace the form with a centred message:
  - Heading (large, 16.5px, weight 600): "Lien invalide ou expiré" / "Invalid or expired link"
  - Body (14.5px, `--muted`): "Demandez un nouveau lien à votre administrateur." / "Ask your administrator for a new link."
  - Button (`.btn-out`): "Retour à la connexion" / "Back to sign in" — links to `/login`
- **Success behaviour:** on POST success → `redirect('/login?invited=1')` (or `?reset=1`); the /login page reads the query and fires a Sonner success toast with copy "Mot de passe défini. Connectez-vous." / "Password set. You can sign in now." then strips the query (`router.replace('/login')`) so refresh doesn't re-toast.

| State | Visual treatment |
|-------|------------------|

| Server error (auth fail on login) | inline error banner above the submit button (NOT a toast — D-22 anti-enumeration) |

Inline field error messages bind to the input via `aria-describedby={`${name}-error`}` and `aria-invalid="true"` so screen readers announce the error.

Validation timing:
- Email field: validate on blur (format check) + on submit (server-side credential check)
- Password field: validate on submit only (no character-class enforcement on login — login accepts whatever the user types and lets Better Auth reject)
- Invite/reset password fields: validate on blur (length + class) AND on change for the strength meter; confirm field validates on blur (match check)

## Accessibility Floor (NON-NEGOTIABLE)

Every interactive surface declared above MUST satisfy:

- **Keyboard reach:** every action is reachable via Tab; no keyboard trap except inside open modals (intentional, modal-spec)
- **Visible focus:** Phase 5 teal box-shadow ring on every focusable element — never `outline: none` without an explicit `:focus-visible` replacement
- **ARIA labels:** every icon-only button has `aria-label` (theme toggle, locale toggle, eye-toggle on password, modal X-button)
- **Live regions:** the inline login error message uses `role="alert"` so screen readers announce it on appearance
- **Form association:** every `<input>` is wrapped in `<label htmlFor>` or referenced by `aria-labelledby`; error messages bind via `aria-describedby` and `aria-invalid="true"`
- **Color contrast:** text on `--surface` is `--ink` (#41423d on #ffffff = 8.6:1 ✓ AAA); text on `--paper` (footer muted) is `--muted` (#6e7191 on #f5f7fc = 4.6:1 ✓ AA); error text on tinted background passes 4.5:1; buttons (`.btn-green` white on #129657) pass 4.5:1 — all verified from Phase 5 token spine
- **Reduced motion:** all transitions wrapped in `@media (prefers-reduced-motion: no-preference)` — modal fade-in, dropdown slide, toast slide all respect the user's preference
