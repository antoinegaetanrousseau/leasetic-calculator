# Phase 5 — UI Design Contract: Bootstrap & Deploy

> Visual and interaction contract for Phase 5. This phase is an infra bootstrap, not a feature phase.
> The only user-facing artifacts are a placeholder `app/page.tsx` and `app/layout.tsx`.
> **Primary purpose of this UI-SPEC is to lock the design-token spine that Phases 6–10 all inherit.**
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

---

## Scope Clarification

Phase 5 produces:

1. `app/layout.tsx` — root layout with font bootstrap, dark-mode class, locale cookie, Tailwind tokens
2. `app/page.tsx` — minimal placeholder: Leasétic logo + brand color + Plus Jakarta Sans + theme toggle + locale toggle (FR/EN)
3. `/healthz` route — JSON-only health probe, no UI

Actual feature screens (login, proposal form, proposals list, admin) are out of scope. This UI-SPEC defines the CSS custom property layer, typography scale, color palette, spacing scale, and theme contract that every subsequent phase will consume — extracted from v10 as the source of truth.

## Theme Bootstrap Contract (BOOT-08, SHELL-07, SHELL-08)

The no-flash inline script in `app/layout.tsx` `<head>` MUST run before the first paint. It reads the `lt_theme` cookie and applies `data-theme` to `<html>`:

```ts
const NO_FLASH_SCRIPT = `
(function(){
  try{
    var m=document.cookie.match(/(?:^|;\\s*)lt_theme=([^;]+)/);
    var t=m?m[1]:'system';
    if(t==='system'){t=window.matchMedia('(prefers-color-scheme:dark)').matches?'dark':'light';}
    document.documentElement.setAttribute('data-theme',t);
  }catch(_){}
})();
`;
```

- Cookie name: `lt_theme`
- Cookie values: `light` | `dark` | `system`
- Default: `system` (resolves via `prefers-color-scheme` media query)
- Server-side: root layout reads cookie and sets `<html data-theme={t}>` for SSR — zero flash even on slow networks
- Tailwind v4 config: `darkMode: ['selector', '[data-theme="dark"]']` (class strategy variant for data attribute)
- Storage: cookie only (NOT localStorage) so SSR renders correct theme on first paint

## Spacing Scale

Declared values (multiples of 4 only, sourced from v10 measurements):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps (`gap:.3rem`→`gap:4px` effective), inline padding, dot indicators |
| sm | 8px | Compact element gaps, nav item icon-to-text gap (`.6rem` ≈ `10px`, round to `8px`) |
| md | 16px | Default element spacing, sidebar padding unit |
| lg | 24px | Section padding (`1.5rem`), main content padding |
| xl | 28px | Card padding (`.card` uses `1.75rem = 28px`) |
| 2xl | 32px | Major section breaks |
| 3xl | 48px | Large gaps between layout regions |
| 4xl | 64px | Topbar height (`--topbar-h: 64px`), page-level spacing |

Exceptions:
- Topbar height: exactly 64px (`--topbar-h`)
- Sidebar width expanded: exactly 260px (`--shell-sidebar-w`)
- Sidebar width collapsed: exactly 72px (v10 `body.shell-collapsed`)
- Footer height: exactly 48px (`--footer-h`)
- Touch targets: minimum 44px height for interactive elements on mobile
- Card padding: 28px (1.75rem) — preserved from v10 `.card` to maintain visual density

## Typography

All sizes sourced from v10 CSS. Base font: **Plus Jakarta Sans**.

| Role | Size | Weight | Line Height | Notes |
|------|------|--------|-------------|-------|
| Body | 14.5px | 400 | 1.55 | `body` rule in v10; main readable text |
| Label / Caption | 11.2px (0.78rem) | 500 | 1.4 | Form field labels, card section titles |
| Small / Muted | 10.5px (0.72–0.75rem) | 400 | 1.5 | Footer text, metadata, timestamps |
| Topbar title | 16.5px (1.15rem) | 600 | 1.2 | Page/section heading in topbar |

Typography constraints:
- Body text: 14.5px at weight 400, line-height 1.55 — matches v10 exactly; do NOT change to 16px or it regresses from v10
- Section titles (`.ctitle`): 11.8px (0.82rem), weight 700, `letter-spacing: 0.06em`, `text-transform: uppercase`, color `var(--muted)`
- Button text: 11.2px (0.78rem), weight 600
- No more than 4 distinct font sizes active on any single screen
- Weights used: 400 (regular) and 600 (semibold) for UI; 700 only for section labels and headings

## Focus Ring Contract (PROP-08, SHELL-11)

Two focus ring states exist. Both are `box-shadow` rings with 3px spread, no `outline`:

| State | Border color | Box shadow | Used for |
|-------|-------------|------------|---------|
| Default input focus | `var(--teal)` = `#2d7a8c` | `0 0 0 3px rgba(45,122,140,0.12)` | All non-error inputs on `:focus` |
| Validation error (on blur) | `var(--danger)` = `#dc2626` | `0 0 0 3px rgba(220,38,38,0.12)` | `input.invalid` — invalid inputs after blur |

Implementation:
- `outline: none` on all inputs
- `border-radius: 12px` on all text/number/password/email/tel inputs
- The error ring class is `invalid` applied to the `<input>` element (not the wrapper)
- For duration-picker buttons (`.db`) in error state: outer wrapper gets `invalid` class with `border: 1.5px solid var(--danger)` and `background: rgba(220,38,38,0.04)` — individual buttons inherit `border-color: var(--danger)`

## Print / PDF CSS Invariants (established Phase 5, enforced Phase 8)

These rules MUST be present in `app/globals.css` from Phase 5 even though PDF is not generated until Phase 8:

```css
@media print {
  @page { size: A4; margin: 0; }
  * { -webkit-print-color-adjust: exact !important; print-color-adjust: exact !important; }
  body { display: block !important; }
  /* sidebar, topbar, footer are display:none in print — add selectors in Phase 6 */
}

/* PDF/print surface is always white — never themed */
html[data-theme="dark"] .prop-page,
html[data-theme="dark"] [data-pdf-surface] {
  background: #fff !important;
  color: #1a2832 !important;
}
```

The `[data-pdf-surface]` data attribute is the v1.1 equivalent of v10's `.prop-page` — use it to mark the proposal preview area in Phase 8.
