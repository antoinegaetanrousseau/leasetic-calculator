```typescript
'use client';

/**
 * D-10-13/14: Yellow informational banner on the coefficients page.
 *
 * Visible when the latest global_params row's coefficients are byte-equal to
 * seedParams.coefficients (i.e., admin hasn't customized yet — still on D-D1
 * placeholder values). The visibility check is performed SERVER-SIDE in
 * `coefficients/page.tsx` (single source of truth — no client comparison) and
 * passed as the `visible` prop. Banner disappears once admin saves any edit
 * (which creates a new global_params row that no longer JSON.stringify-matches
 * seedParams.coefficients).
 *
 * Closes the long-standing TODO at src/lib/calc/seed-params.ts ("// TODO:
 * confirm against v10 baseline before CUT-06") — the verification mechanism
 * IS this banner; the TODO is resolved by data (admin saves an edit), not by
 * a code change.
 *
 * Accessibility: role="status" + aria-live="polite" — screen readers announce
 * the banner on page load without stealing focus (banner is informational,
 * not interactive).
 */
import { t, type Lang } from '@/lib/i18n/dictionaries';

interface SeedBannerProps {
  lang: Lang;
  visible: boolean;
}

export function SeedBanner({ lang, visible }: SeedBannerProps) {
  if (!visible) return null;
  return (
    <div
      className="seed-banner"
      role="status"
      aria-live="polite"
    >
      {t('admin.seed_banner.message', lang)}
    </div>
  );
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Build artifact → browser bundle | NEXT_PUBLIC_PRIVACY_URL_* values are inlined at build time and visible in the client JS bundle |
| Login page → third-party privacy URL | Link target opens in a new tab; window.opener access must be denied |
| Server component → client island | isStillSeed is computed server-side; SeedBanner trusts the prop without re-validating |
| seedParams (build-time constant) → server-side comparison | seedParams is imported as a JS module — not user-supplied; no injection surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-04-01 | Tampering | malicious env var (NEXT_PUBLIC_PRIVACY_URL_FR=javascript:...) injecting an XSS into the login page | mitigate | Env vars are operator-controlled (not user-supplied). Risk surface is "Antoine misconfigures Vercel Production env." Defense in depth: hardcoded fallback only fires if env var unset; an INVALID URL value would NOT cross the type gate (string assigned to `href` attribute is escaped by React). Operator hygiene is the primary control |
| T-10-04-02 | Spoofing | reverse-tabnabbing — privacy URL opens in new tab, gets `window.opener` access to login session | mitigate | `rel="noopener noreferrer"` blocks `window.opener` access; `target="_blank"` is paired with the rel attributes. Acceptance criterion enforces both attributes |
| T-10-04-03 | Information disclosure | NEXT_PUBLIC_PRIVACY_URL_* values inlined into client bundle | accept | URLs are public (privacy policy IS publicly hosted). No confidentiality concern. NEXT_PUBLIC_ prefix is the documented Next.js mechanism for build-time-public values |
| T-10-04-04 | Information disclosure | broken link if env var unset and no fallback | mitigate | Hardcoded fallback URLs (`https://leasetic.fr/mentions-legales`, `https://leasetic.fr/privacy-policy`) match `.env.example` defaults from 10-01. Deployment with env var unset still produces a working link |
| T-10-04-05 | Tampering | client-side seed-vs-current comparison could be bypassed by a malicious client | mitigate | The deep-equal happens SERVER-SIDE in coefficients/page.tsx (line ~40). The client SeedBanner only renders the prop it's given. A malicious client could prevent the banner from rendering but cannot fake "still on seed values" to a different visitor — comparison authority is the server |
| T-10-04-06 | Information disclosure | seedParams.coefficients values are inlined into server bundle and could leak via SSR HTML | accept | seedParams contains the placeholder D-D1 baseline values. These are NOT confidential — they're the publicly-known v10 fixture coefficients (committed in plain in src/lib/calc/seed-params.ts, visible in any git clone). No secret value crosses the boundary |
| T-10-04-07 | Repudiation | banner false-negative — admin re-saves the SAME values as seedParams (e.g., to add a note) and banner disappears even though they didn't actually customize | accept | This is the documented limit of byte-equal comparison (D-10-14). The mitigation is operator hygiene: if admin saves the exact seed values intentionally, the banner disappears because Phase 9 created a new global_params row with effective_from > seed row's effective_from — that NEW row has a notes field carrying the admin's review. The TODO closure (D-10-14 by data) accepts this as the "good enough" definition of "verified" |
| T-10-04-08 | Repudiation | banner false-positive — banner renders even when admin has saved a customized row, because deep-equal matches by coincidence | accept | Mathematical lower bound: probability of randomly customizing a 4×3 coefficient table to JSON.stringify-equal seedParams is effectively zero. If admin DOES want to "customize back to seed values," the documented behavior is "banner re-appears" which is accurate |
| T-10-04-09 | Elevation of privilege | non-admin user reaches /[adminSegment]/coefficients and sees seedParams via banner | mitigate | The page is gated by `requireAdmin()` at line 27 (independent of layout) per AUTH-15. Non-admin sessions are redirected before the page renders. SeedBanner only ships in the admin tree |
| T-10-04-10 | Spoofing | login page reachable to anyone — privacy link href is influenced by `lang` (which is reachable via cookie / search param) | accept | Both langs resolve to a hardcoded fallback if env var unset. The langs are the only two values supported (`'fr' | 'en'`) — no untrusted lang injection |
