## Area 2 — Seed coefficients data source (CALC-08, DATA-12 alignment)

The only coefficients in v10 source are the made-up fixture coeffs in `assertCalc()`. So "seed v10 baseline values" requires Antoine to *provide* the canonical set OR extract from a known-good partner's localStorage. **Options presented:**

3. Use fixture coeffs in Phase 7, defer real values to Phase 8 — zero coupling but live preview in dev shows numbers that don't match what partners would have seen in v10.

**User selection:** TS constant in `lib/calc/seed-params.ts` (Recommended).

**Notes / rationale:**
- Single source of truth feeds Phase 8's seed migration without duplication.
- Type safety catches drift at compile time.
- Placeholder discipline (`// TODO: confirm against v10 baseline before CUT-06`) accommodates the fact that canonical values aren't yet in hand.

## Area 3 — Calc module shape (CALC-01)

**Question:** Given seed-params is its own file, how should the rest of `src/lib/calc/` be split?

**Pre-question insight surfaced:** Phase 5 scaffolded `src/lib/calc/.gitkeep` as a directory, suggesting the original intent was a multi-file split. CALC-01's literal wording ("module at `src/lib/calc.ts`") is a logical reference, not a single-file mandate. The previous answer (D-2) committed `src/lib/calc/seed-params.ts` as a path, which already implies directory.

**Options presented:**
1. Minimal split (Recommended) — `index.ts` (barrel) + `seed-params.ts` + `schema.ts` + test file. 3 production files.
2. Granular split — `index.ts` + `seed-params.ts` + `schema.ts` + `tranche.ts` + `coefficients.ts` + `formula.ts` + test. 6 files. Maps cleanly to v10's mental model.
3. Flat single file — `src/lib/calc.ts` only; conflicts with D-2 unless backtracked.

**User selection:** Granular split.

**Notes / rationale:**
- Each file stays ≤ ~80 lines; barrel re-exports keep import paths clean.
- `tranche.ts` / `coefficients.ts` / `formula.ts` separation maps to v10 mental model and makes targeted unit tests trivial.
- Public consumers (form, future Phase 8 server route, future `lib/pdf`) only ever import from `@/lib/calc` (the barrel).

**Locked as:** D-3 in 07-CONTEXT.md.

## Area 4 — Numeric type discipline (CALC-08, PITFALLS §3.5 alignment)

**Question:** What numeric-type discipline should Phase 7's calc engine adopt at the boundary?

**Pre-question insight surfaced:** PITFALLS §3.5's "Decimal not Number" rule is specifically about Postgres `numeric` ↔ JS round-trip drift (Drizzle returns numeric() as strings; JS auto-coerces on math; errors compound). Phase 7 has no Postgres. For the magnitudes here (<10M €, 4-decimal coeffs), IEEE 754 doubles are exact through ~15 significant digits, so the formula `a × (1 + comm/100) × coeff / 100` has zero observable drift in pure JS. The discipline matters when persistence shows up (Phase 8).

**Options presented:**
1. Number internal + string seam (Recommended) — strings at both ends of the calc engine boundary (Postgres-numeric-compatible from day 1); internally parses to Number, math, formats back. Phase 8 swaps NO signatures.
2. Decimal library now — adds `decimal.js` / `big.js` runtime dep. Heavy and probably premature.
3. Number everywhere, defer to Phase 8 — pragmatic; Phase 8 will need to refactor signatures when DB round-trip lands.

**User selection:** Number internal + string seam (Recommended).

**Notes / rationale:**
- Lightest discipline that survives Phase 8 — the calc engine's interface is already DB-string-compatible.
- Precision discipline lives in the *type contract*, not in a runtime dep.
- Single `parseNumeric` / `formatNumeric` helper in `lib/calc/formula.ts` (or imports from `lib/db/numeric.ts` if Phase 5 established that file — planner verifies).

**Locked as:** D-4 in 07-CONTEXT.md.
