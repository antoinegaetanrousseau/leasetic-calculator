### 1.4 Open question resolved (STATE.md Open Q2)

**Q2:** "Existing v10 form schema — does it already capture a structured 'client name' field, or only LC reference text input?"

**Answer (from reading v10 HTML lines 545–551):** **v10 already captures TWO structured client fields, not just LC reference:**

- `<input id="client-co">` — "Société cliente" (client *company* name) — line 546, placeholder "Raison sociale"
- `<input id="client-name">` — "Nom du destinataire" (*recipient person* name) — line 547, placeholder "Prénom Nom"
- Plus 4 ancillary recipient fields: `client-role`, `client-tel`, `client-email`, `client-siren` (all already structured in v10)

The LC reference (`LC-XXXXX`) is a server-generated random string (v10 line 1741: `'LC-' + Math.floor(Math.random() * 90000 + 10000)`), NOT a form input — it's auto-generated at "generate" time and surfaced via the copy-LC button.

**PROP-06 resolution:** PROP-06 requires "a required `client_name` field for the home-page list." In v10 this field already exists as `client-co` (Société cliente / Client company name) — that's the natural list-display identifier (the business, not a specific person). Phase 7 promotes `client-co` from "soft-required" (v10 toast on blank) to "required by Zod schema" with an inline blur-time error message.

Phase 7 keeps BOTH `client-co` AND `client-name` as separate structured fields per v10 parity (PROP-06 says "all v10 inputs plus a required `client_name` field" — interpreted as "all v10 inputs are preserved verbatim AND `client_name`/`client-co` is required"). The home-page list (Phase 8) will display `client-co` as the headline identifier. Per-proposal recipient details (`client-name`, `client-role`, `client-tel`, `client-email`, `client-siren`) live on the detail page (Phase 8) and inside the PDF.

**No NEW field is added in Phase 7 vs v10.** The "delta" called out in REQUIREMENTS.md PROP-06 ("plus a required client_name field") is satisfied by tightening the v10-existing `client-co` field to required-by-schema. This decision is logged in §10 Decisions.

## 5. Pattern Contracts

### 5.1 Blur validation with red-ring focus pattern (PROP-08)

**Rule:** RHF mode = `'onBlur'`. Each field validates onBlur via the shared Zod schema. `shouldFocusError: true` on submit. `.invalid` class application is handled by RHF's `aria-invalid="true"` ↔ CSS selector `[aria-invalid="true"]` mapping (preferred over manual class toggling — keeps semantics + visual in sync).

**Implementation contract (Phase 7 plan must satisfy):**

```css
/* in globals.css — verify present, add if missing */
input.invalid,
input[aria-invalid="true"],
.ieu:has(input[aria-invalid="true"]) {
  border-color: var(--danger) !important;
  box-shadow: 0 0 0 3px rgba(220, 38, 38, 0.12) !important;
}

.dg.invalid,
.dg[aria-invalid="true"] {
  border-color: var(--danger) !important;
  box-shadow: 0 0 0 3px rgba(220, 38, 38, 0.12) !important;
}
```

**Inline error message:**
- Element: `<p role="alert" id="{fieldId}-error" class="error-msg">`
- Position: directly under the input, margin-top 4px (xs)
- Style: 11.2px weight 500, color `--danger`, line-height 1.4
- Bound to input via `aria-describedby="{fieldId}-error"` and only rendered when `errors[fieldId]?.message` is truthy
- Hidden when valid (do NOT keep an empty placeholder line — that creates layout shift; instead use `display: none` when no error, accept the 4px+text-height layout shift on appearance — match v10 behavior)

**Animation (optional, respects prefers-reduced-motion):**
- Default: no animation on `.invalid` ring appearance (instant transition)
- If `@media (prefers-reduced-motion: no-preference)`: 150ms ease transition on `border-color` and `box-shadow` (already in §3.2.4 `transition: border-color 150ms, box-shadow 150ms`)

**Server-side parity (CALC-04, SHELL-11):**
- The Zod schema lives in `src/lib/proposal/schema.ts` and is imported on BOTH client (RHF resolver) AND server (when Phase 8's POST /api/proposals adds a server-side parse). Phase 7 ships the schema even though no server endpoint exists yet — the file goes in shared lib so Phase 8 can pick it up unchanged.

### 5.2 Live-preview update timing (PROP-07)

**Rule:** Update on every `<form>` value change (RHF `useWatch` subscription) BUT debounced internally to 300ms to avoid layout-jank during fast typing.

**Pattern:**

```ts
const watched = useWatch({ control, name: ['amount', 'duration', 'validityDays'] });
const debounced = useDebouncedValue(watched, 300); // 300ms debounce
const computed = useMemo(() => computeLoyer(debounced), [debounced]);
```

**Why debounce, not throttle:** The user types continuously; debounce ensures the preview only re-computes once the user pauses (300ms). Throttle would re-compute every 300ms during continuous typing, making the preview "flicker" on fast typers.

**Why 300ms:** v10 ran update on every `input` event (no debounce). Phase 7 picks 300ms as a sensible default — fast enough to feel "live" (perceptual threshold ≈ 100ms; 300ms still feels responsive), slow enough to skip 90% of intermediate states during typing. If user testing shows sluggishness, planner can tune to 150ms.

**Initial render:** No debounce on first render — `computed` is computed synchronously from default form values (idle state because amount=0 < 25001 and duration=null).

**Computation source:** `src/lib/calc.ts:computeLoyer()` (CALC-02, ported from v10's `calcRent()`). Same module on both client and server (CALC-07: client preview only, server recomputes on save).

### 5.3 Number / currency input affordances

**Amount input:**
- `inputMode="numeric"` (mobile keyboards show numeric pad)
- `type="text"` (so we can format with U+202F narrow no-break space)
- `autoComplete="off"`
- Format on input: digit-only via regex; thousand-separator inserted right-anchored; cursor stays at end (acceptable UX trade-off; v10 does this)
- Accept paste of formatted strings (e.g., "75 000" → strip spaces → re-format)

**Display vs storage:**
- Display: formatted with U+202F narrow no-break space ("75 000")
- Storage in form state: plain integer (`75000`)
- Validation in Zod: `.transform(s => parseInt(s.replace(/\D/g,''), 10)).refine(n => n > 25000)`

**Currency suffix:** "€ HT" — explicitly bilingual? **No** — "€ HT" reads identically in FR and EN (HT = "Hors Taxes" / "exclusive of VAT"; in B2B EN it's commonly "ex-VAT" but "HT" is universally understood in EU B2B contexts). v10 line 572 uses "€ HT" in both FR and EN (key `common.ht` resolves to "&euro; HT" in BOTH languages — already in dictionaries.ts). Keep this pattern.

**Locale formatters (per Phase 6 D-28):**
- All formatted output (live-preview loyer, future PDF amounts) uses `lib/i18n/format.ts:formatCurrency(amount, lang)` which calls `Intl.NumberFormat('fr-FR' | 'en-GB', { style: 'currency', currency: 'EUR' })` explicitly.
- French formatting: `1 234,56 €` (narrow no-break space, comma decimal)
- English formatting (en-GB): `€1,234.56` (Intl.NumberFormat with currency:'EUR' and locale 'en-GB' produces this)
- Coefficient display: `r.coeff.toFixed(4)` + " %" (per v10 line 1773); Phase 7 uses the same — 4 decimal places, narrow no-break space before `%` per French typography convention (use `formatNumber` helper from Phase 6).

## 6. Layout & Composition Decisions

### 6.1 Why two-column form layout

v10 uses a `.two` class with `grid-template-columns: 1fr 1fr` (lines 567 onwards) to put project-params on the right and partner/client/interests cards on the left. v10 also embeds the live preview ("Aperçu") *inside* the project-params card.

Phase 7 deviates slightly: split the preview OUT of the project-params card and into its own sticky right column. Rationale:

1. **Sticky preview is more useful than embedded:** Users scroll through the form (especially on tablet sizes); a sticky preview keeps the loyer visible during scroll. v10's approach worked because v10 was a single non-scrolling page.
2. **LC ref + validity selector live in the same space:** Both are *outputs/properties of the proposal* rather than form inputs — co-locating them in the preview card preserves conceptual coherence.
3. **Visual hierarchy:** The loyer is the most important number on the page; sticky placement gives it permanent visual prominence.

Trade-off: the project-params card on the left becomes pure-input (no preview), so the project-params card visually balances the partner/client cards above it. This also matches the React Hook Form mental model where one form scope = one column.

### 6.2 Why 300ms debounce on live preview

See §5.2. Decision logged in §10.

### 6.3 Why no error toast for inline-validation blur events

Inline validation on blur (per-field) → inline `<p role="alert">` error message, no toast.
Submit-attempt with errors → Sonner error toast "Vérifiez les champs requis." (ONE toast, not per-field).

Rationale: inline messages are spatially co-located with the offending field (low cognitive cost); toasts are summary signals (high cognitive cost per fire). Firing a toast per blur would be overwhelming; firing inline error is the standard pattern.

### 6.4 Why no real-time form-state persistence (e.g., localStorage draft)

Phase 7 does NOT auto-save form state to localStorage. PITFALLS §10.8 explicitly cautions against the "save on every keystroke" mental model carryover from v10. Phase 8 will introduce the explicit "Générer" → DB save flow. If a user navigates away mid-form, they lose their inputs — acceptable for v1.1 (Phase 7 is a 5-minute interaction; v1.2+ can add draft persistence if user complaints surface).

## 13. Accessibility Floor (NON-NEGOTIABLE)

Carries forward Phase 6's NON-NEGOTIABLE list, plus Phase-7-specific:

- **Keyboard reach:** every action reachable via Tab. Form Tab order: partner-co → partner-name → client-co → client-name → client-role → client-tel → client-email → client-siren → SLB Yes → SLB No → Eval Yes → Eval No → amount → duration buttons (36/48/60) → project-desc → partner-ref → reset → generate → preview validity (15/30/60) → copy LC button. Linear, predictable, skipping nothing.
- **Visible focus:** `--teal` 3px box-shadow ring on every focusable element. NO `outline: none` without an explicit `:focus-visible` replacement. (Inherited from Phase 5/6.)
- **ARIA labels:** every icon-only button has aria-label. Specifically for Phase 7:
  - Copy button: `aria-label="Copier la référence LC-12345"` (or interpolated current ref) for screen-reader clarity
  - Yes/No toggles: each button gets `role="radio"` with `aria-checked` reflecting state; wrapper gets `role="radiogroup"` + aria-label of the question
  - Duration / Validity segmented: same — `role="radiogroup"` on wrapper, `role="radio"` + `aria-checked` on buttons
- **Live regions:** the form-level submit error toast uses Sonner's built-in `role="status"`; the inline field error messages use `role="alert"` (announces on appearance)
- **Form association:** every `<input>` is wrapped in `<label htmlFor>` or referenced by `aria-labelledby`. Error messages bind via `aria-describedby` and `aria-invalid="true"`.
- **Required marker accessibility:** `*` is purely visual; the `aria-required="true"` attribute is the source of truth for assistive tech.
- **Color contrast:** all Phase 5 + Phase 6 contrast ratios verified; Phase 7's new tokens (none — pure inheritance) inherit the same ratios. Specifically: `--danger` (`#dc2626`) on `--surface` (`#ffffff`) = 4.5:1 ✓ AA for the inline error messages.
- **Reduced motion:** all transitions wrapped in `@media (prefers-reduced-motion: no-preference)`. The 150ms input border/ring transitions; 2-second copy-button label switch happens instantly (no fade) to avoid motion-sensitivity issues.
- **Screen reader announcement of computed loyer:** the live-preview computed-value block has `aria-live="polite"` + `aria-atomic="true"` so the loyer is announced when it changes. Throttle to once per 500ms to avoid annoyance during fast typing.
