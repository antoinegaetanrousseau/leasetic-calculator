### 6.2 `<Stepper>` (COMP-01)

**File:** `src/components/ui/Stepper.tsx`
**Server component:** yes (Phase 13 wizard derives `currentStep` + `completedSteps` from URL pathname server-side and passes as props).
**Figma nodes:** stepper area at top of `35:46` (step 1 active), `39:46` (step 1 done + step 2 active), `40:46` (steps 1+2 done + step 3 active).

#### Prop signature
```ts
export interface StepperProps {
  /** Which step is the "current" step (highlighted as active). */
  currentStep: 1 | 2 | 3;
  /** Array of step numbers that have been completed (render with check mark, clickable). */
  completedSteps: number[]; // e.g., [1, 2] when currentStep === 3
  /** i18n language — required for step labels via t(). */
  lang: Lang;
  /** Optional override for step labels. Defaults: ['proposals.wizard.step1', 'proposals.wizard.step2', 'proposals.wizard.step3']. */
  stepLabels?: [string, string, string];
  /** Optional href builder. If provided, completed steps wrap in <Link href={hrefForStep(n)}>. If undefined, completed steps render as non-interactive <span>. */
  hrefForStep?: (step: 1 | 2 | 3) => string;
}
```

#### Step state derivation rule
For each step `n ∈ {1, 2, 3}`:
- if `completedSteps.includes(n)` AND `n !== currentStep` → state `done`
- else if `n === currentStep` → state `active`
- else → state `pending`

**Invariant:** `currentStep` MUST NOT appear in `completedSteps`. Plan validation: render-time `console.warn` (dev only) if violated, but the derivation rule above still produces correct output (active wins).

#### DOM contract (per step, in order 1 → 2 → 3, with connector lines between consecutive steps)

Container: `<ol>` with `role="list"`, padding `20px 28px` (vertical / horizontal), background `var(--surface)`, border `1px solid var(--border)`, borderRadius `16px`, boxShadow `var(--shadow-card)`. Displays as `flex; align-items: center; gap: 12px;`.

Each `<li>` is a flex item, `display: flex; align-items: center; gap: 8px;`.

Between `<li>` items, a connector line: `<span aria-hidden="true">` with `flex: 1; height: 2px; background: var(--border);` (renders as a 2px horizontal rule filling the gap).

**Step circle markup (per state):**

| State | Element | Background | Border | Inner content | Inner color |
|---|---|---|---|---|---|
| `active` | `<span class="stepper-circle stepper-circle--active">` | `var(--gd)` | none | `{step number}` (text) | `#ffffff` |
| `done` | `<span class="stepper-circle stepper-circle--done">` | `var(--gd)` | none | `<Check size={16} strokeWidth={2.5} aria-hidden="true" />` (lucide-react) | `#ffffff` |
| `pending` | `<span class="stepper-circle stepper-circle--pending">` | `var(--paper)` | `2px solid var(--border)` | `{step number}` (text) | `var(--muted)` |

All three circles: `width: 32px; height: 32px; border-radius: 9999px; display: inline-flex; align-items: center; justify-content: center; flex-shrink: 0; font-size: 14px; font-weight: 600; font-family: var(--font-sans);`.

**Step label markup (per state):**
| State | Element | Color | Weight | aria attributes |
|---|---|---|---|---|
| `active` | `<span>` | `var(--ink)` | 600 | `aria-current="step"` (on the parent `<li>`) |
| `done` | `<Link>` if `hrefForStep` provided, else `<span>` | `var(--ink)` | 600 | when `<Link>`: standard link semantics; when `<span>`: no extra aria |
| `pending` | `<span>` | `var(--muted)` | 500 | `aria-disabled="true"` (on the parent `<li>`) |

All labels: `font-size: 14.5px; line-height: 1.55; font-family: var(--font-sans);`.

#### Clickability contract
| State | Wrapper | Cursor | Behavior on click |
|---|---|---|---|
| `done` AND `hrefForStep` provided | `<Link href={hrefForStep(n)}>` | `pointer` | Next.js client-side navigation |
| `done` AND `hrefForStep` omitted | `<span>` (NOT a button, NOT a link) | `default` | no-op |
| `active` | `<span>` | `default` | no-op |
| `pending` | `<span>` with `aria-disabled="true"` | `default` (NOT `not-allowed` — Figma shows neutral) | no-op |

**Focus-visible** on a `<Link>` (done state): box-shadow `0 0 0 3px rgba(45,122,140,0.18)` on the **step circle** (i.e., the visual focus indicator wraps the 32×32 green circle, not the whole `<li>`). Match existing `.btn-green:focus-visible` pattern from `app/globals.css`.

#### Acceptance criteria (Stepper)
- AC-ST-01: `<Stepper currentStep={1} completedSteps={[]} lang="fr" />` renders: step 1 active green circle with text "1", steps 2+3 pending outlined circles with text "2"/"3" and muted labels.
- AC-ST-02: `<Stepper currentStep={2} completedSteps={[1]} lang="fr" />` renders: step 1 done green circle with check icon, step 2 active green circle with text "2", step 3 pending outlined circle.
- AC-ST-03: `<Stepper currentStep={3} completedSteps={[1,2]} lang="fr" />` renders: steps 1+2 done with check, step 3 active green circle with text "3".
- AC-ST-04: Without `hrefForStep`, no `<a>` elements render (all steps are `<span>`); verified by `queryAllByRole('link')` returning 0.
- AC-ST-05: With `hrefForStep={(n) => '/proposals/new/' + ['parametres','calcul','verification'][n-1]}`, done steps render `<a>` with the correct href; pending step renders `<span>` (no `<a>`); active step renders `<span>` (no `<a>`).
- AC-ST-06: Connector line between two consecutive `<li>` items has computed `height: 2px` and `background-color` equal to `rgb(217, 219, 233)` (light mode `--border`) or `rgb(36, 44, 62)` (dark mode).
- AC-ST-07: Active step's parent `<li>` has `aria-current="step"`. Pending step's parent `<li>` has `aria-disabled="true"`.
- AC-ST-08: Step number font: computed `font-size: 14px; font-weight: 600`. Step label font: computed `font-size: 14.5px`. Label weight 600 when active/done, 500 when pending.
- AC-ST-09: Container has computed `padding: 20px 28px; border-radius: 16px; background-color` matching `--surface`.
- AC-ST-10: Tab key in a fresh-loaded `/proposals/new/calcul` (Phase 13 consumer) focuses step 1's link first (since it's the only `done`-clickable step in that state); focus ring is teal at 3px spread.

```html
  </footer>

```

**`{sidebar-width}` in `grid-template-columns`:** must respond to the sidebar's collapsed/expanded state. Two implementation options for the planner:
1. **Option A (simpler, preserves SSR):** Use `var(--shell-sidebar-w)` always (260px). When sidebar collapses, it sets its own width to 72px and the grid column STAYS at 260px, leaving a 188px gutter. **Rejected** — visually broken per Figma.
2. **Option B (canonical):** The sidebar's collapsed state must shift the grid column. Implementation: a CSS variable `--shell-sidebar-current-w` that the client `RetractableSidebar` component updates on `<html>` or `<body>` via `document.documentElement.style.setProperty('--shell-sidebar-current-w', collapsed ? '72px' : '260px')` on mount + toggle. The grid uses `grid-template-columns: var(--shell-sidebar-current-w) 1fr`. First-paint SSR: `--shell-sidebar-current-w` defaults to `260px` (declared in `:root` in `app/globals.css`). One-frame layout shift on initial render for collapsed users (same trade-off as RetractableSidebar's own width state).

**Decision:** Option B is canonical for Phase 11. Planner specifies the exact DOM-property write location (on mount + on toggle).

### 11.2 `.chip-expired` rewrite breaks v1.1 visual contract (regression risk)

See §6.6 "Class-naming collision flag". Plan must include a smoke task: open `/` (partner home) with at least one expired proposal in seed data, confirm the new muted-gray chip renders cleanly and is distinguishable from active/draft.

### 11.3 LocaleToggle + ThemeToggle full-width modifier

The existing components in `src/components/LocaleToggle.tsx` (lines 10–14) and `src/components/ThemeToggle.tsx` (lines 17–21) hard-code `inline-flex items-center` + a fixed `p-1` padding via Tailwind classes. To make them full-width inside the sidebar's expanded layout, two paths:

- **Path A:** Add an optional `fullWidth?: boolean` prop. When true, the wrapper switches to `display: flex; width: 100%;` and each segment gets `flex: 1` for equal distribution. Smaller diff, preserves single source of truth.
- **Path B:** RetractableSidebar renders its own copy of the toggle markup (DOM duplication), calling the same Server Actions (`setLang`, `setTheme`) but with sidebar-specific styling. Larger diff, but isolates sidebar chrome from topbar use case.

**Recommendation (planner to confirm):** Path A. Smaller diff, single source of truth.

### 11.4 RetractableSidebar SSR collapsed-state flash

See §6.3 hydration rule. Phase 11 ships with the one-frame flash for collapsed users. Phase 13 may upgrade to a cookie-driven SSR collapse if user feedback requires (analogous to `lang` + `theme` cookies). Not a blocker for Phase 11 ship.

### 11.5 `--shell-sidebar-current-w` CSS variable write site

Decision (§6.7 "Shell" Option B): the `<RetractableSidebar>` client component writes `--shell-sidebar-current-w` to `document.documentElement.style` on mount and on toggle. Risk: writing to `<html>` style attribute is a shared global side-effect. Alternative: write to `<aside>` itself + use a CSS sibling selector on `<aside>`. Planner picks; document choice in plan SUMMARY.

### 11.6 Admin sidebar nav hrefs need `adminSegment`

See §6.3 nav-item set (admin). Since `RetractableSidebar` is a client component, it cannot directly read `process.env.ADMIN_URL_SEGMENT` (server-only). The `<Shell>` server component reads it via the (admin) layout's `params.adminSegment` (already available) and passes the resolved hrefs as a prop to `RetractableSidebar`. Pattern: `<RetractableSidebar adminHrefs={{home: \`/\${adminSegment}\`, coefficients: \`/\${adminSegment}/coefficients\`, ...}} />`. Concrete prop name + shape decided by planner.
