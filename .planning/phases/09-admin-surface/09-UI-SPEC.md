### 3.1.2 Explain-calculation Section

Placement: a `<section>` rendered **between** the editor card and the history card (not a tab — simpler to implement, no URL state needed). The section is a `.card` wrapper with a clearly distinct heading so the admin can scroll past it.

**Why a section not a tab:** Tab UI requires URL state (`?tab=explain`) or client-side state, and adds complexity for a surface that is supplementary. The section layout lets the admin see the editor form values above while scrolling to the explain tool below — a tight feedback loop.

#### 3.1.2.3 Explain tool output — formula trail

Rendered as a `<div>` below the input row, appearing immediately when all three inputs have valid values. Output is structured HTML (not a `<pre>` block):

```
┌─────────────────────────────────────────────────────────┐
│  FORMULE                                                │
│  loyer = montantHT × (1 + commission/100) × coeff / 100│
│                                                         │
│  SUBSTITUTION                                           │
│  = 50 000 × (1 + 5/100) × 2.3000 / 100                │
│  = 50 000 × 1.05 × 2.3000 / 100                       │
│  = 1 207.50 €/mois                                     │
│                                                         │
│  Tranche : T2 · Durée : 48 mois · Validité : 30 jours  │
└─────────────────────────────────────────────────────────┘
```

**Visual spec:**

Container: `background: var(--paper); border: 1px solid var(--border); border-radius: 12px; padding: 20px 24px; margin-top: 16px; font-family: ui-monospace, 'Cascadia Code', 'Fira Mono', monospace; font-size: 13px; line-height: 1.7; color: var(--ink)`

Section label (e.g. "FORMULE"): `font-family: var(--font-sans); font-size: 10.5px; font-weight: 700; letter-spacing: 0.1em; color: var(--muted); text-transform: uppercase; margin-bottom: 6px`

Formula string line: `color: var(--muted)` (symbolic, gray — it is the named-variable form)

Substitution lines: `color: var(--ink)` (numeric substitution steps)

**Final result line** (the loyer): `font-size: 18px; font-weight: 700; color: var(--gd); font-family: var(--font-sans)` — formatted via `formatCurrency(loyerHT, lang)`.

`commission_pct` in the formula trail: rendered as its numeric value (e.g. `5`) in the substitution steps — no special marking, no redaction. This is the ONLY non-editor surface where it appears by design (ADMIN-09).

**"On demand" state:** when amount exceeds `max_amount`, render a muted pill: `{t('result.sur.demande')}` (reuse existing key) in `--gold` color, 14px / 600. No formula trail.

**Missing inputs state:** before all three inputs are filled, render a placeholder `<p style="color: var(--muted); font-style: italic;">` = `{t('admin.coefficients.explain.placeholder')}` = "Renseignez le montant et la durée pour simuler le calcul." / "Enter amount and duration to simulate the calculation."

#### 3.1.2.4 Explain tool interaction contract

- Compute is **reactive** (not behind a "Calculate" button) — updates on every input change via React state / `useMemo(computeLoyer, [inputs])`.
- Zero server calls — pure-client `computeLoyer()` imported from `@/lib/calc`.
- No loading state (compute is synchronous).
- No sonner toast.
- No audit_log write.

### 4.3 Disable / Re-enable Confirmation

**Decision: sonner confirm-toast pattern, NOT a modal.**

Rationale: disable/re-enable is a low-stakes reversible action (admin can always re-enable). A full modal adds unnecessary friction. A toast with undo is the appropriate UX pattern for reversible admin actions.

**Implementation:**

On "Désactiver / Disable" button click:
1. Fire sonner `toast.info()` immediately (before the server action) with a built-in confirm pattern:
   ```
   {t('admin.accounts.toast.disable.confirm')} = 
     "Désactiver {displayName} ? Le partenaire ne pourra plus se connecter."
     / "Disable {displayName}? The partner will not be able to log in."
   ```
   Duration: 6 seconds (long enough to read + act)
   **Note:** standard Sonner does not have a native confirm-toast with buttons. Implement as a plain sonner action-toast (using Sonner's `action` API if available, or a custom toast component): show message + "Confirmer / Confirm" and "Annuler / Cancel" inline buttons inside the toast.

2. If admin clicks "Confirmer" inside the toast: server action fires, partner disabled, row status chip updates via `revalidatePath`.

3. If admin clicks "Annuler" or toast times out: no action, no server call.

4. On server action success: sonner success: `{t('admin.accounts.toast.disable.success')}` = "{displayName} désactivé(e)." / "{displayName} disabled."

5. On server action error: sonner error: `{t('admin.accounts.toast.disable.error')}` = "Erreur lors de la désactivation." / "Error disabling account."

**Re-enable flow:** same pattern. Confirm copy: `admin.accounts.toast.enable.confirm` = "Réactiver {displayName} ?" / "Re-enable {displayName}?" Success: `admin.accounts.toast.enable.success` = "{displayName} réactivé(e)." / "{displayName} re-enabled."

**Interaction contract:** while a disable/re-enable server action is in-flight, the row's action buttons are disabled (spinner) per §3.2.6.

## 12. Accessibility Contract

- **Table semantics:** both the coefficient table and the accounts table use native `<table>` / `<thead>` / `<tbody>` / `<tr>` / `<th>` / `<td>` elements. `<th scope="col">` on column headers, `<th scope="row">` on tranche label cells in the coefficient table.
- **Modal focus trap:** both modals (save confirmation, create partner) trap focus inside the panel while open. On close, focus returns to the trigger button.
- **Modal ARIA:** `role="dialog" aria-modal="true" aria-labelledby="{titleId}"`.
- **Action buttons:** each per-row action button has `aria-label="{action} {displayName}"` (e.g., `aria-label="Désactiver Antoine R."`) to disambiguate identical action labels across rows.
- **Explain tool:** `aria-live="polite"` on the formula trail container, so screen readers announce when the result updates.
- **Status chips:** `role="status"` not needed — chips are static text inside table cells. Screen reader reads the text naturally.
- **Search input:** `aria-label="{t('admin.accounts.search.placeholder')}"` on the search input (no visible `<label>` tag — label is the placeholder).
- **Keyboard interaction:** all modals are dismissible via Escape key (maps to "Annuler / Cancel"). Tab order within modals follows DOM order.
- **Loading buttons:** disabled buttons during loading use `aria-disabled="true"` and `aria-busy="true"` on the button, in addition to the visual `opacity: 0.6`.

## 13. Commission Invisibility Lockdown (ADMIN-09)

This section is a **cross-cutting contract** that every executor and reviewer must honor.

**`commission_pct` is visible in exactly 2 places on Phase 9 surfaces:**

| Surface | Context | How it appears |
|---------|---------|---------------|
| Coefficients editor card | Admin edits it | Input field, labeled "Commission apporteur (%)" |
| Explain tool formula trail | Admin verifies math | Numeric value in substitution steps (e.g., `5` in `(1 + 5/100)`) |

**`commission_pct` is ABSENT from:**
- Accounts/partners list page — no commission data at all
- History table metadata row (timestamp / admin / note columns)
- History table "changes" cell — **commission_pct DOES appear here** when it was changed (it IS one of the fields that can change), but ONLY in the diff format `commission_pct: 5.00 → 5.50`. This is the expected behavior per ADMIN-09 (the audit trail must include commission changes for operational transparency; ADMIN-09 prohibits *unprompted* commission display, not commission in a user-initiated diff view).
- Save-confirmation modal diff — **commission_pct DOES appear here** if changed (same rationale as history diff — it is a changed-field diff that the admin explicitly triggered by clicking Save).
- Admin home page
- Any route handler response JSON (except the `global_params` endpoint consumed by the editor)
- Server-side console/log statements (logger discipline)
- Network responses for the accounts list, history metadata, or any other admin endpoint

**Verification checklist for executor:**
- [ ] `/api/admin/accounts` or equivalent endpoint: response JSON does NOT include `commission_pct`
- [ ] History table query result: history row metadata displayed in the table does NOT include raw `commission_pct` value as a column — it appears only inside the rendered diff cell when it was the changed field
- [ ] Audit log records for `user.disable`, `user.create`, etc.: payload shape does NOT include `commission_pct`
- [ ] Server actions other than `adminUpdateGlobalParams`: no logging of `commission_pct`
