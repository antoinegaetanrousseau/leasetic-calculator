## Issues encountered + how they were resolved

- **Premature `</script>` close in assertEscape fixture** â€” A literal `</script>` string inside the JS test fixture closed the HTML script block early, dumping unprocessed JS as visible page content. Fixed by escaping the forward slash (`<\/script>`) in the fixture string. Discovered via Antoine's screenshot on first browser test.
- **Phase 1 PARITY-21 gap** â€” The spec implied a confirm dialog on RĂ©initialiser but neither v9 nor v10 had one. Deferred from Phase 1 audit to Phase 3 plan UX-02, which shipped the confirm.
- **CSS `--sidebar-w` collision** â€” v9's existing `--sidebar-w: 24px` was used by the proposal page's internal navy strip. Phase 4 introduced `--shell-sidebar-w: 260px` instead of overloading the existing variable.
- **Duplicate IDs from topbar-vs-inline button consolidation** â€” Phase 4 needed Save/Lock buttons in the topbar AND inline in the admin panel. Topbar versions renamed to `btn-save-topbar` / `btn-lock-topbar` to avoid duplicate `id=` attributes; both delegate to the inline button's click handler.
- **Dashly icon swap rejected, reverted** â€” Mid-milestone iteration where the feather-style line icons were temporarily swapped for filled-green Dashly tile icons. Antoine preferred the original feather style; reverted in commit `dcbd70b` while keeping all retractable-sidebar work intact (clean separation made the surgical revert possible).

## Architecture patterns established (carry forward to v1.1+)

- **Planâ†’Summary 1:1 ratio as the milestone-completeness invariant.** The close workflow checks `find ... -name '*-SUMMARY.md' | wc -l` against the plan count.
- **CONTEXT.md as the durable phase decision log.** Locked decisions go here before planning starts. Plans implement, summaries record outcomes.
- **On-load self-checks (`assertCalc`, `assertEscape`, `assertValidity`) as the regression early-warning system.** Every page load logs N/N green or red. Future maintainers know immediately if something broke.
- **CSS sections with labeled comment dividers** (TOKENS / LAYOUT / COMPONENTS / PRINT). New rules go in their semantic section.
- **JS sections with labeled comment dividers** (STATE / I18N / UTILS / CALC / ADMIN / PROPOSAL / MIGRATION / INIT). New code goes in its semantic section.
- **Sequential waves for shared-file projects.** Wave dependencies are sequential because plans share `Matrice_2026_THE_Leasetic-v10.html`.
- **Atomic commits per task.** Every task is its own commit so any single change is revertable.
- **`<symbol>` + `<use>` sprite for icons.** Single source of truth; `currentColor` inheritance lets icons adapt to surrounding state for free.
- **Tokens, then surfaces, then states.** New design changes start by introducing/renaming a CSS variable, then changing the surfaces that use it, then handling state-specific overrides last.
